## Overview

GoModel can forward client headers to upstream providers, strip headers you do not
want to send upstream, and inject fixed headers for every request to a provider.

## Configuration

```yaml
providers:
  openrouter:
    type: openrouter
    api_key: ${OPENROUTER_API_KEY}
    passthrough_user_headers: true
    passthrough_user_headers_skip_mode: allow
    passthrough_user_headers_skip:
      - "X-Internal-Trace"
    custom_upstream_headers:
      X-Title: "MyApp"
      User-Agent: "MyApp/1.0.0"
```

### Fields

| Field | Type | Description |
|---|---|---|
| `passthrough_user_headers` | boolean | When `true`, client headers that are not skipped are forwarded upstream. |
| `passthrough_user_headers_skip_mode` | `skip` \| `allow` | In `skip` mode, the listed headers are dropped. In `allow` mode, only the listed headers are forwarded. |
| `passthrough_user_headers_skip` | list of strings | Header names acted on by `passthrough_user_headers_skip_mode`. Case-insensitive. |
| `custom_upstream_headers` | map of strings | Headers injected on every request to this provider. Values support `${ENV_VAR}` substitution. |

<Warning>
  `passthrough_user_headers` defaults to `false`; when it is `false`, no user headers
  are forwarded. When `passthrough_user_headers` is `true` and `passthrough_user_headers_skip_mode` is `skip`
  or `""` (default), an empty `passthrough_user_headers_skip` list forwards all user headers except the
  hard-coded blocklist. When `passthrough_user_headers` is `true` and `passthrough_user_headers_skip_mode` is
  `allow`, an empty `passthrough_user_headers_skip` list forwards no user headers.
</Warning>

<Warning>
  Transport headers such as `Accept-Encoding`, `Connection`, `Transfer-Encoding`,
  and `Keep-Alive` are always blocked. GoModel manages the HTTP transport itself;
  attempting to override these headers has no effect.
</Warning>

## Header Hierarchy and Precedence

For each request, headers are applied in this order:

1. **Static `custom_upstream_headers`** — injected first, from `config.yaml`
2. **`passthrough_user_headers`** — client-supplied headers forwarded on top

When the same header name appears in both, the later source wins:

- Static `custom_upstream_headers` for `X-Title` → `X-Title` sent upstream with that value
- Caller passthrough header for `X-Title` → caller value overrides the static one

## Hard-Blocked Headers

The following headers are **never forwarded**, even if listed in `custom_upstream_headers` or `passthrough_user_headers`:

- `authorization`, `proxy-authorization`, `cookie`, `set-cookie`
- `x-api-key`, `api-key`, `x-goog-api-key`, `x-auth-token`, `x-access-token`, `x-gomodel-key`
- `X-GoModel-User-Path` and the configured user-path header alias
- Transport/hop-by-hop: `Accept-Encoding`, `Connection`, `Keep-Alive`, `TE`, `Transfer-Encoding`, `Trailer`, `Upgrade`, `Content-Length`, `Host`

## Environment Variable Overrides

Per-provider env vars override the corresponding `config.yaml` values:

| Env var | Overrides | Format |
|---|---|---|
| `<PROVIDER>_CUSTOM_UPSTREAM_HEADERS` | `custom_upstream_headers` | Comma-separated `key=value` pairs |
| `<PROVIDER>_PASSTHROUGH_USER_HEADERS` | `passthrough_user_headers` | Boolean (`true`/`false`) |
| `<PROVIDER>_PASSTHROUGH_USER_HEADERS_SKIP` | `passthrough_user_headers_skip` | Comma-separated header names |
| `<PROVIDER>_PASSTHROUGH_USER_HEADERS_SKIP_MODE` | `passthrough_user_headers_skip_mode` | `skip` or `allow` |

Examples:

```bash
# OpenAI
OPENAI_CUSTOM_UPSTREAM_HEADERS="X-Title=MyApp, User-Agent=MyApp/1.0.0"
OPENAI_PASSTHROUGH_USER_HEADERS_SKIP="X-Internal-Trace, X-Debug"

# OpenRouter (inherits OpenAI-compatible CompatibleProvider behavior)
OPENROUTER_CUSTOM_UPSTREAM_HEADERS="X-Title=MyApp"
OPENROUTER_PASSTHROUGH_USER_HEADERS=true
```

Whitespace around entries is ignored.

## Modes

### Skip mode

Drop the listed headers. When `passthrough_user_headers: true` and `passthrough_user_headers_skip` is
empty, all user headers except the hard-coded blocklist are forwarded.

```yaml
providers:
  deepseek:
    type: deepseek
    api_key: ${DEEPSEEK_API_KEY}
    passthrough_user_headers: true
    passthrough_user_headers_skip_mode: skip
    passthrough_user_headers_skip:
      - "X-Internal-Trace"
      - "X-Debug"
    custom_upstream_headers:
        User-Agent: "MyApp/1.0.0"
```

### Allow mode

Only forward headers that appear in `passthrough_user_headers_skip`. Use this when you want an
explicit allowlist. When `passthrough_user_headers_skip` is empty, no user headers are forwarded.

```yaml
providers:
  groq:
    type: groq
    api_key: ${GROQ_API_KEY}
    passthrough_user_headers: true
    passthrough_user_headers_skip_mode: allow
    passthrough_user_headers_skip:
      - "X-Title"
      - "User-Agent"
    custom_upstream_headers:
        X-Title: "MyApp"
        User-Agent: "MyApp/1.0.0"
```

## OpenRouter Identity Headers

OpenRouter uses `HTTP-Referer` and `X-OpenRouter-Title` to attribute your app on
their site. GoModel sets these from env vars when targeting an `openrouter` provider:

| Env var | Upstream header |
|---|---|
| `OPENROUTER_SITE_URL` | `HTTP-Referer` |
| `OPENROUTER_APP_NAME` | `X-OpenRouter-Title` |

These are applied as **env-derived static headers**, placed in the same layer as
`custom_upstream_headers`. Precedence at runtime:

1. `OPENROUTER_SITE_URL` / `OPENROUTER_APP_NAME` env vars set a header value
2. Static `custom_upstream_headers` in `config.yaml` override that value
3. Caller passthrough headers (via `passthrough_user_headers: true`) override both

### Concrete scenarios

| Scenario | `HTTP-Referer` sent upstream | `X-OpenRouter-Title` sent upstream |
|---|---|---|
| Only `OPENROUTER_SITE_URL` set | `OPENROUTER_SITE_URL` value | — (blocked) |
| Only `OPENROUTER_APP_NAME` set | — (blocked) | `OPENROUTER_APP_NAME` value |
| Both `OPENROUTER_SITE_URL` and `OPENROUTER_APP_NAME` set | `OPENROUTER_SITE_URL` value | `OPENROUTER_APP_NAME` value |
| Both env vars set, plus `custom_upstream_headers` for `X-OpenRouter-Title: MyApp` | `OPENROUTER_SITE_URL` value | `MyApp` (static wins over env) |
| All three above, plus caller sends `X-OpenRouter-Title: CallerApp` | `OPENROUTER_SITE_URL` value | `CallerApp` (caller wins over static and env) |

## Limitations

Header overrides apply only to providers that use the OpenAI-compatible
`CompatibleProvider` adapter:

- [`azure`](/providers/azure)
- [`bailian`](/providers/bailian)
- [`deepseek`](/providers/deepseek)
- `fireworks`
- `groq`
- `minimax`
- [`ollama`](/providers/multiple-ollama)
- [`kimicode`](/providers/kimicode)
- `openai`
- [`opencode-go`](/providers/opencode-go)
- `openrouter`
- [`oracle`](/providers/oracle)
- [`vllm`](/providers/vllm)
- `xai`
- [`xiaomi`](/providers/xiaomi)
- `zai`

The feature does **not** apply to `anthropic`, `gemini`, `vertex`, or `bedrock`
providers because they use provider-specific adapters rather than the
OpenAI-compatible path. Prose references to those providers in examples is fine,
but configuration of these header fields under those provider blocks is ignored.
