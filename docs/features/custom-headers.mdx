## Overview

GoModel can forward client headers to upstream providers, strip headers you do not
want to send upstream, and inject fixed headers for every request to a provider.

This page describes the configuration fields, the header precedence rules, and how
the feature interacts with provider-specific default headers such as OpenRouter's
identity attribution headers.

## Configuration

```yaml
providers:
  openrouter:
    type: openrouter
    api_key: ${OPENROUTER_API_KEY}
    passthrough_user_headers: true
    passthrough_user_headers_skip_mode: skip
    passthrough_user_headers_skip:
      - "X-Internal-Trace"
    custom_upstream_headers:
      X-Title: "MyApp"
      User-Agent: "MyApp/1.0.0"
```

### Fields

| Field | Type | Description |
|---|---|---|
| `passthrough_user_headers` | boolean | When `true`, client headers that are not skipped are forwarded upstream. Defaults to `false`. |
| `passthrough_user_headers_skip_mode` | `skip` \| `allow` | In `skip` mode, the listed headers are dropped. In `allow` mode, only the listed headers are forwarded. |
| `passthrough_user_headers_skip` | list of strings | Header names acted on by `passthrough_user_headers_skip_mode`. Case-insensitive. |
| `custom_upstream_headers` | map of strings | Headers injected on every request to this provider. Values support `${ENV_VAR}` substitution. |

<Warning>
  When `passthrough_user_headers` is `false`, no user headers are forwarded.
  When it is `true` and `passthrough_user_headers_skip_mode` is `skip` or empty,
  an empty `passthrough_user_headers_skip` list forwards all user headers except the
  hard-coded blocklist. When it is `true` and `passthrough_user_headers_skip_mode` is
  `allow`, an empty `passthrough_user_headers_skip` list forwards no user headers.
</Warning>

<Warning>
  Transport headers such as `Accept-Encoding`, `Connection`, `Transfer-Encoding`,
  and `Keep-Alive` are always blocked. GoModel manages the HTTP transport itself;
  attempting to override these headers has no effect.
</Warning>

## Header hierarchy and precedence

For each upstream request, headers are applied in four layers. Later layers override
earlier layers for the same header name.

| Order | Layer | Source | Can it be overridden? |
|---|---|---|---|
| 1 | Auth headers | Injected by the provider adapter, for example `Authorization: Bearer …` | No |
| 2 | Provider default headers | Values the provider requires GoModel to send, such as OpenRouter's `HTTP-Referer` and `X-OpenRouter-Title` | Yes, by layer 3 or 4 |
| 3 | Static `custom_upstream_headers` | Fixed headers from `config.yaml` or the matching `<PROVIDER>_CUSTOM_UPSTREAM_HEADERS` env var | Yes, by layer 4 |
| 4 | Passthrough user headers | Client-supplied headers forwarded when `passthrough_user_headers: true` | Highest priority |

A later layer always wins for the same header name. For example:

- A provider default `HTTP-Referer` is sent unless `custom_upstream_headers` also sets `HTTP-Referer`.
- A static `HTTP-Referer` is sent unless the caller also passes `HTTP-Referer` through with `passthrough_user_headers: true`.

## Provider default headers

Some providers ship with default headers that GoModel attaches automatically. These
defaults are resolved at provider construction time and are not exposed as a
`config.yaml` field. They sit between auth headers and your static custom headers,
so you can override them with `custom_upstream_headers` or caller passthrough headers.

### OpenRouter

OpenRouter uses two identity headers to attribute traffic on the OpenRouter dashboard:

| Env var | Upstream header | Purpose |
|---|---|---|
| `OPENROUTER_SITE_URL` | `HTTP-Referer` | App URL shown in the OpenRouter dashboard |
| `OPENROUTER_APP_NAME` | `X-OpenRouter-Title` | App name shown in the OpenRouter dashboard |

GoModel reads these env vars and attaches them as provider default headers. They are
not configurable through `config.yaml`; they are only set from the runtime environment.

<Note>
  OpenRouter also accepts `X-Title` from callers, but the dashboard app name is
  driven by `X-OpenRouter-Title`. Use `X-OpenRouter-Title` when you want to
  override the displayed app name.
</Note>

### Skip-list behavior with provider defaults

The `passthrough_user_headers_skip` list only filters **caller** headers. It does not
strip provider default headers. If a caller sends a header that matches a provider
default (for example, `HTTP-Referer`), and that header name is listed in
`passthrough_user_headers_skip`, the caller's value is dropped but the provider's
default value is still attached.

To replace a provider default, set it explicitly in `custom_upstream_headers`. To let
a caller replace it, do not list it in `passthrough_user_headers_skip` and enable
`passthrough_user_headers: true`.

### Concrete scenarios for OpenRouter

| Scenario | `HTTP-Referer` sent upstream | `X-OpenRouter-Title` sent upstream |
|---|---|---|
| Only `OPENROUTER_SITE_URL` set | `OPENROUTER_SITE_URL` value | `GoModel` (default app name) |
| Only `OPENROUTER_APP_NAME` set | `https://gomodel.enterpilot.io` (default site URL) | `OPENROUTER_APP_NAME` value |
| Both env vars set | `OPENROUTER_SITE_URL` value | `OPENROUTER_APP_NAME` value |
| Both env vars set, plus `custom_upstream_headers` with `X-OpenRouter-Title: MyApp` | `OPENROUTER_SITE_URL` value | `MyApp` |
| All of the above, plus caller sends `X-OpenRouter-Title: CallerApp` | `OPENROUTER_SITE_URL` value | `CallerApp` |
| Both env vars set, caller sends `HTTP-Referer`, and the name is in `passthrough_user_headers_skip` (skip mode) | `OPENROUTER_SITE_URL` value | `OPENROUTER_APP_NAME` value |

## Hard-blocked headers

The following headers are **never forwarded**, even if listed in `custom_upstream_headers` or `passthrough_user_headers`:

- `authorization`, `proxy-authorization`, `cookie`, `set-cookie`
- `x-api-key`, `api-key`, `x-goog-api-key`, `x-auth-token`, `x-access-token`, `x-gomodel-key`
- `X-GoModel-User-Path` and the configured user-path header alias
- Transport/hop-by-hop headers: `Accept-Encoding`, `Connection`, `Keep-Alive`, `TE`, `Transfer-Encoding`, `Trailer`, `Upgrade`, `Content-Length`, `Host`

## Environment variable overrides

Per-provider env vars override the corresponding `config.yaml` values:

| Env var | Overrides | Format |
|---|---|---|
| `<PROVIDER>_CUSTOM_UPSTREAM_HEADERS` | `custom_upstream_headers` | Comma-separated `key=value` pairs |
| `<PROVIDER>_PASSTHROUGH_USER_HEADERS` | `passthrough_user_headers` | Boolean (`true`/`false`) |
| `<PROVIDER>_PASSTHROUGH_USER_HEADERS_SKIP` | `passthrough_user_headers_skip` | Comma-separated header names |
| `<PROVIDER>_PASSTHROUGH_USER_HEADERS_SKIP_MODE` | `passthrough_user_headers_skip_mode` | `skip` or `allow` |

Examples:

```bash
OPENROUTER_CUSTOM_UPSTREAM_HEADERS="X-OpenRouter-Title=MyApp, User-Agent=MyApp/1.0.0"
OPENROUTER_PASSTHROUGH_USER_HEADERS=true
OPENROUTER_PASSTHROUGH_USER_HEADERS_SKIP="X-Internal-Trace"
```

Whitespace around entries is ignored.

## Modes

### Skip mode

Drop the listed headers. When `passthrough_user_headers: true` and the skip list is
empty, all user headers except the hard-coded blocklist are forwarded.

```yaml
providers:
  deepseek:
    type: deepseek
    api_key: ${DEEPSEEK_API_KEY}
    passthrough_user_headers: true
    passthrough_user_headers_skip_mode: skip
    passthrough_user_headers_skip:
      - "X-Internal-Trace"
      - "X-Debug"
    custom_upstream_headers:
      User-Agent: "MyApp/1.0.0"
```

### Allow mode

Only forward headers that appear in the skip list. Use this when you want an explicit
allowlist. When the skip list is empty, no user headers are forwarded.

```yaml
providers:
  groq:
    type: groq
    api_key: ${GROQ_API_KEY}
    passthrough_user_headers: true
    passthrough_user_headers_skip_mode: allow
    passthrough_user_headers_skip:
      - "X-Title"
      - "User-Agent"
    custom_upstream_headers:
      X-Title: "MyApp"
      User-Agent: "MyApp/1.0.0"
```

## Supported providers

Header overrides apply only to providers that use the OpenAI-compatible
`CompatibleProvider` adapter:

- [`azure`](/providers/azure)
- [`bailian`](/providers/bailian)
- [`deepseek`](/providers/deepseek)
- `fireworks`
- `groq`
- `minimax`
- [`ollama`](/providers/multiple-ollama)
- [`kimicode`](/providers/kimicode)
- `openai`
- [`opencode-go`](/providers/opencode-go)
- `openrouter`
- [`oracle`](/providers/oracle)
- [`vllm`](/providers/vllm)
- `xai`
- [`xiaomi`](/providers/xiaomi)
- `zai`

The feature does **not** apply to `anthropic`, `gemini`, `vertex`, or `bedrock`
providers because they use provider-specific adapters rather than the
OpenAI-compatible path. Configuration of these header fields under those provider
blocks is ignored.
